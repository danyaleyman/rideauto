# Аудит репозитория (полный обзор, не только UI)

Срез по подсистемам: что в хорошем состоянии после работ по цепочкам каталога, **что остаётся вне UI** для уровня зрелого продакшена.

## Сильные стороны

| Область | Состояние |
|---------|-----------|
| Каталог API | Контракт v1/v2, golden, рантайм-валидация, кэш с эпохой, гидратация Postgres |
| Поиск | Meilisearch, фильтры, сортировки, drift фасетов, дедуп-ключ в индексе + `distinctAttribute` |
| Наблюдаемость API | Prometheus: latency, счётчики, кэш, размер ответа (при материализованном body), enrich LLM |
| Инфраструктура | Docker Compose, sync-скрипт, preflight gate, документация блоков J+K / I / L / M / N |
| Безопасность процессов | Gitleaks в CI, секреты через env |

## Пробелы и риски (помимо доработок UI)

### Данные и консистентность

- **Postgres ↔ Meili**: задержка синка; при сбое воркера выдача может отставать — нужны алерты на возраст последнего успешного sync / lag документов.
- **Дедуп**: ключ в индексе и `distinctAttribute` схлопывают выдачу; **каноническая одна строка на VIN в Postgres** всё ещё не внедрена (слияние дублей, `inactive`) — см. `BLOCK_L_DEDUP.md`.
- **Оценка total при distinct**: уточнить на проде поведение `estimatedTotalHits` и курсорной пагинации.

### Наблюдаемость и SLO

- **Alerting**: метрики есть, но **готовые правила Prometheus/Grafana** в репозитории не зафиксированы — вынести пороги p95 / 5xx / размер ответа в IaC или доку `BLOCK_M_SCALE_COST.md`.
- **Tracing**: распределённый трейс (OpenTelemetry) для цепочки search → PG не подключён.
- **Логи**: структурированные корреляционные id для инцидентов — по желанию.

### Безопасность и злоупотребления

- **Rate limiting** публичных `/api/*` — проверить на edge (Cloudflare) и при необходимости на API.
- **Аудит внутренних маршрутов** (`/api/internal/*`) — секреты, IP allowlist в проде.

### CI / качество

- **CI** (`.github/workflows/ci.yml`): убедиться, что прогон покрывает backend + при необходимости web lint/build.
- **E2E** против реального Meili/Postgres в CI обычно нет — приёмочные тесты на стейдже.

### Воркеры и пайплайны

- Скраперы, ночной sync, чекеры sold — **мониторинг exit code и длительности** job’ов вне репозитория (cron/systemd).
- **Идемпотентность** повторных импортов и восстановление после частичного сбоя — задокументировать runbook по каждому воркеру.

### Правовое / операционное

- Ретеншн персональных данных (лиды, auth) — политики вне кода.
- Бэкапы Postgres и восстановление — вне репозитория.

## Вывод

Цепочка **«скрап / Postgres → Meili → API → Next»** для каталога **доведена до production-разумного уровня** в коде контракта, кэше, фасетах и дедупе в индексе. До «взрослого» операционного уровня не хватает в основном **автоматизированных алертов, SLO, серверного слияния дублей в БД и приёмочных проверок total/пагинации** после включения distinct — это **не UI**, а платформа и данные.
