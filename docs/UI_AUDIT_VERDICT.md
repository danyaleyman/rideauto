# Аудит UI и вердикт (план доработок)

Дата: 2026-05-02. Объект: публичный фронтенд `web` (Next.js 15, каталог, карточка авто, заявки, юридические страницы).

## Краткий вердикт

Продукт **визуально и структурно уже близок к аккуратному B2C**: единая типографика (Inter), сдержанная палитра, осмысленные отступы, русскоязычный контент, формы с валидацией и юридическими отсылками. До уровня «как у крупных зрелых команд» не хватает в основном **инфраструктуры качества вокруг UI** (регрессии, метрики, доступность под прицелом, дизайн-токены и процесс), а не разового «докрасить пиксели».

## Что уже на хорошем уровне

- **Навигация и иерархия**: главная → каталог → карточка → заявка; страницы политик и cookie.
- **Формы**: явное согласие на ПДн, ссылки на `/privacy`, отдельный блок согласия с контрастной обводкой (после правок блока J).
- **Cookie-баннер**: фикс снизу, `safe-area`, ссылки на политики выделены, `role="dialog"` для скринридеров.
- **Каталог**: фильтры, URL-состояние, клиентский поиск — типичный паттерн маркетплейса.
- **Smoke-покрытие**: Playwright на мок-API + unit-тесты на чистые функции (каталог URL, цены, VIN).

## Разрывы относительно «взрослых» крупных проектов

### 1. Дизайн-система и консистентность

- **Токены**: часть оттенков задана ad-hoc (`ring-black/[0.04]` и т.п.) — у крупных продуктов обычно единая сетка spacing/radius/color semantic tokens и документация для дизайнеров.
- **Состояния**: не везде явно проработаны loading / empty / error / offline на уровне всего каталога (не только `error.tsx`).
- **Компонентный каталог**: Storybook или аналог для критичных блоков (карточка, фильтры, диалоги) — снижает регрессии при рефакторинге.

### 2. Доступность (a11y)

- Фокус-ловушки в модалках, порядок табуляции, связь `aria-describedby` с подсказками — имеет смысл **аудит по WCAG 2.1 AA** (axe в CI + ручная проверка форм).
- Контраст ссылок в «приглушённых» зонах — периодически перепроверять на светлой/тёмной теме.

### 3. Производительность и UX-сигналы

- **Web Vitals** уже есть компонент отчёта — следующий шаг: бюджеты в CI (Lighthouse CI или аналог), отслеживание LCP на карточке и каталоге.
- **Изображения**: политика размеров `next/image`, приоритет LCP на герое карточки — выверить по реальным данным.
- **Длинные списки**: виртуализация при росте выдачи (если планируется «бесконечный» скролл без пагинации).

### 4. Надёжность и наблюдаемость

- Клиентские `fetch` с retry — хорошо; для «взрослого» уровня добавляют **корреляцию с бэкендом**, Sentry (или аналог) с релизами и source maps, пользовательские жалобы через ID сессии/запроса.
- **Feature flags** для рискованных UI-изменений — типичная практика на масштабе.

### 5. Контент и доверие

- **Социальное доказательство**: отзывы, кейсы, цифры по доставкам — часто вынесены в отдельные блоки с фото/видео (без изменения юридического смысла текущих страниц).
- **Микрокопирайтинг** ошибок и пустых состояний — единый тон и понятные следующие шаги (написать в Telegram, сбросить фильтры).

### 6. Процесс и регрессии

- **Визуальные снапшоты** (Chromatic, Percy, Loki) для 5–10 ключевых экранов.
- **E2E в CI** на моках (уже заложено) + периодический прогон против стейджа.
- **Локализация**: если планируются языки, закладывать i18n-ключи заранее, а не строки в разметке.

## Приоритизированный план (рекомендация)

| Приоритет | Действие |
|-----------|----------|
| P0 | Держать зелёными `test:unit` и `test:e2e` в CI; при падении стейджа — не блокировать юнит-тесты. |
| P1 | Axe + ручной проход форм/модалок; исправить критичные нарушения фокуса и подписей. |
| P1 | Бюджеты LCP/CLS на главной, каталоге и карточке; алерт при регрессии. |
| P2 | Storybook (или аналог) для карточки каталога, фильтров, диалога быстрой заявки. |
| P2 | Визуальные снапшоты для статичных страниц (buy, privacy, cookies). |
| P3 | Унификация дизайн-токенов и dark/light проверка контраста по палитре. |

## Итог

Текущий UI **достаточен для честного запуска и доверия пользователя**, особенно с учётом юридически аккуратных форм и cookie. Чтобы выглядеть и вести себя «как у крупных игроков», основной упор стоит сделать на **измеримость (Core Web Vitals, ошибки), доступность и автоматические регрессии**, а не на бесконечное усложнение визуала.
