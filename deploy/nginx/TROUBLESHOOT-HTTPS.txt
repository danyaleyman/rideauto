Сайт «не открывается» в браузере, а curl на 127.0.0.1:80 с Host даёт 200
============================================================================

1) Слушает ли nginx порт 443
   sudo ss -tlnp | grep ':443'
   Если пусто — в конфиге нет listen 443 ssl (часто certbot в отдельном файле, который отключили).

2) Есть ли server на 443 для rideauto.ru
   sudo nginx -T 2>/dev/null | grep -nE "listen 443|ssl_certificate|server_name.*rideauto"

3) Проверка TLS с самого сервера
   curl -sI -H "Host: rideauto.ru" https://127.0.0.1/ -k | head -15
   -k отключает проверку имени в сертификате (для 127.0.0.1 это нормально).

4) Сертификаты certbot
   sudo certbot certificates
   Смотрите Domains и Expiry. Продление:
   sudo certbot renew --dry-run

5) Срок и цепочка сертификата для домена (с сервера)
   echo | openssl s_client -servername rideauto.ru -connect 127.0.0.1:443 2>/dev/null | openssl x509 -noout -dates -subject

6) Файрвол
   sudo ufw status
   При активном ufw нужно: sudo ufw allow 'Nginx Full'   или allow 80,443/tcp

7) DNS (с сервера или с ПК)
   dig +short rideauto.ru A
   Должен быть публичный IP вашего VPS.

8) Браузер
   Попробуйте окно инкогнито или другой браузер; отключите VPN.
   Ошибка NET::ERR_CERT_* — смотреть п.4–5.

Восстановление HTTPS после «слёта» сертификата (типовой путь)
-------------------------------------------------------------
sudo certbot --nginx -d rideauto.ru -d www.rideauto.ru
sudo nginx -t && sudo systemctl reload nginx

Важно: deploy/nginx/rideauto.conf в репо описывает только listen 80.
Если вы копируете его поверх sites-available и затираете блок certbot с 443 —
HTTPS пропадёт. Тогда снова certbot --nginx или вручную объедините ssl в один server{}.

Каталог не грузится после certbot (HTTPS), а по HTTP всё ок
============================================================
Симптом: на https://rideauto.ru/catalog пусто / ошибки в консоли на запросы
/api/cars, /api/facets, /api/counts (404, HTML вместо JSON, CORS и т.п.).

Причина: certbot часто добавляет отдельный server { listen 443 ssl; } с минимальным
набором location. В нём нет proxy_pass на 127.0.0.1:8080 для /api/* — запросы API
уходят в try_files статики и ломают каталог.

Что сделать
-----------
1) Посмотреть, что реально отдаётся по HTTPS для API:
   curl -sI -H "Host: rideauto.ru" https://127.0.0.1/api/health -k | head -5

2) Сравнить полный конфиг — есть ли в server 443 те же блоки, что в server 80
   (особенно location /api/cars, /api/facets, /api/ и т.д.):
   sudo nginx -T 2>/dev/null | less
   (найти оба server_name rideauto.ru — для :80 и :443)

3) Исправление: в server { listen 443 ssl ... } для rideauto.ru должны быть
   те же proxy location, что в rideauto.conf (от «location /api/...» до
   «location /api/»), плюс статика (root, gzip, location /, @wra_extensionless).
   Ориентир: deploy/nginx/rideauto.conf и deploy/nginx/snippets/wra-frontend-routes.conf
   + комментарий в rideauto.conf про копирование блоков в 443.

4) После правки: sudo nginx -t && sudo systemctl reload nginx

См. также: deploy/nginx/FIX-DUPLICATE-SERVER.txt — если активен «не тот» server{}.
