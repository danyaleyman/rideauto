# Резервное копирование SQLite (каталог Encar)

## Режим WAL

API и скрейперы обычно включают `PRAGMA journal_mode=WAL`. Рядом с файлом БД появляются:

- `имя.db-wal` — журнал записей;
- `имя.db-shm` — shared memory.

Для **консистентной** копии одного файла `.db` в активной записи:

1. Остановите процессы, которые пишут в БД (API, импорт), **или**
2. Выполните онлайн-бэкап через SQLite (см. ниже).

Копирование только `encar_cars.db` или **`encar_china.db`** (Dongchedi) без `-wal`/`-shm` при активной записи может дать **повреждённый** снимок — для каждого файла каталога выполняйте отдельный `.backup` (см. выше).

## Онлайн-бэкап (рекомендуется)

На сервере, где есть `sqlite3` CLI:

```bash
sqlite3 /path/to/encar_cars.db ".backup '/path/to/backup/encar_cars_$(date +%Y%m%d).db'"
```

Или из Python (кратко): открыть исходную БД и вызвать `backup` API целевого соединения.

## Полное копирование в простое

Если записи нет (сервис остановлен), скопируйте все три файла, если они есть:

- `encar_cars.db` (+ `-wal`, `-shm` при необходимости)
- `encar_china.db` (+ `-wal`, `-shm`) — китайский каталог Dongchedi

## Снапшот тома

На VPS с LVM/снапшотами диска можно делать снимок тома с БД, затем копировать файлы из снапшота — это согласовано на уровне ФС.

## Расписание

Настройте `cron` или systemd timer: ежедневный `.backup` в отдельный каталог + ротация (удаление копий старше N дней).

## Проверка восстановления (рекомендуется раз в квартал)

1. На тестовой машине или временном каталоге разверните копию бэкапа `.db` (и при необходимости `encar_china.db`).
2. Запустите API с этими путями (`--db`, `--db-china`) и выполните `GET /api/health?deep=1`: поля `catalog_db.readable`, `cars_rows` должны быть в порядке.
3. Выборочно откройте `/api/cars?page=1` и `/api/car/{id}` по известному id из бэкапа.
4. Зафиксируйте время прохождения и отклонения (если бэкап не поднялся — исправляйте процесс, а не только файл).
