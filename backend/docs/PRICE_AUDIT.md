# Аудит ценообразования (2026): `backend/pricekorea.py`, `backend/pricechina.py`, `backend/market_pricing_shared.py`

## Что изменено под продукт

1. **Комиссия (Корея и Китай)** — по ступеням от «ценности», а не процент по умолчанию:
   - Корея: база для ступени = **авто + 440 000 ₩ документов в ₽** (как было в смысле `car_and_docs_rub`).
   - Китай: для ступени используется **только юаневая машина в ₽** (`price_cny × курс ЦБ`), без логистики по Китаю и без растаможки в этом пороге. Эффективный процент в ответе — от `(машина_₽ + таможня_итого + брокер)` — см. ключ `commission_rate_effective`.

2. **Фрахт** — `FREIGHT_USD × курс USD ЦБ` (нет строки USD в котировках — запасный `USDT/RUB` с Binance, как указано в коде).

3. **440 000 ₩** и **тело объявления в KRW** — в ₽ по **₩ ЦБ**, если есть `Valute.KRW`; иначе прежний **USDT‑мост** (совместимость и отказоустойчивость).

4. **Брокер `BROKER_RUB = 86_000`** — фиксируется как блок «брокер + СБКТС и документы на таможне» в комментариях модуля; правка суммы только константа/конфиг.

## Риски и несовпадения с «идеальной» финмоделью

- **Смешение источников курсов**: официально вы запросили ЦБ для $ и ₩; **Binance/USDT всё ещё используются** как fallback для ₩ и для USD если нет `USD` в `daily_json` (или при сетевых сбоях). Для клиентских обещаний лучше зафиксировать в оферте: «расчёт на дату по курсам ЦБ; при технической недоступности — котировки рынка».

- **Китай: порог комиссии** не включает `china_docs_delivery` и не включает фрахт (в корейской ветке докорейские доки входят в порог). При необходимости выровнять бизнес‑смысл — расширить первый аргумент `commission_rub_tiered` для ветки Che168/pricechina.

- **Таможенные таблицы** (пошлина, сбор, утиль): эвристики и коэффициенты утиля требуют периодической сверки с ЕТС ЕЭК / ПП РФ по датам ввоза.

- **Физлицо только**: акциз/НДС ввоза нулевые; юрлица и коммерческий ввод не моделируются.

- **`commission_rate_default` в ответах** установлен в `0.0` как маркер «не процентная схема»; процент доступен только как `commission_rate_effective`.

## Рекомендации следующих шагов

1. **Прозрачность в API/каталог**: при необходимости отдавать в JSON разложение `cbr_*` и флаг `krw_pricing_source` уже считаются в результате калькулятора — можно промапить в `pricing_clean`/slim без раздувания публичного API.

2. **Meilisearch на хосте**: `WRA_PG_DSN`/SYNC_PG_DSN с именем контейнера `postgres` с хоста не резолвится — использовать `127.0.0.1` или `compose exec api` (`deploy/scripts/run_meilisearch_sync_host.sh` уже ориентирован на локальный порт Postgres).

3. **HP‑каталог**: при появлении ключей добить `/data/hp_catalog.db`, чтобы перевести больше Encar‑карт в «full_customs».

4. **Тест‑заморозки**: добавить один интеграционный smoke с записанными фикстурами ЦБ/KRW если CI без сети.
